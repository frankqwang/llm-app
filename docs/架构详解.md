# VlogCopilot 架构详解

流水线由单个 `WorkManager` `CoroutineWorker`（`VlogPipelineWorker`）托管，内部由 `PipelineOrchestrator` 驱动 7 个以上阶段。所有状态写入 `filesDir/`，以便 CLI 调试器、应用内 Process 标签页或单元测试重放任意一次运行。

## 完整流程

```
PhotoIngest → EventSegmenter → EventSelectionPlanner（粗排）
                                      │
                                      ▼
                           Gemma 4 E2B-IT 启动一次
                                      │
                                      ▼
                  EventScout（3×3 联系人表 → VLM 信号，
                   每个 top-K 事件的故事弧/人物/节奏）
                                      │
                                      ▼
                  EventSelectionPlanner 精排，选出 top-N 事件
                                      │
                                      ▼
                  选中资产的轻量感知（人脸/清晰度/NSFW——无 LLM，JSON 缓存）
                                      │
                                      ▼
                  VlmAnnotator 逐资产标注（场景/主体/动作/氛围/显著性
                   + 视频最佳时刻索引 + 时间窗口）
                                      │
                                      ▼
                  对每个 top-N 事件执行 runEvent()：
                    Browser  → EventMemory（蒙太奇 VLM 调用）
                    Audience → AudienceBrief + 节奏
                               （紧凑/均衡/悠长）
                    Director → DirectorBrief + 镜头蓝图 + 调色
                               （target_duration 被限制在节奏区间内；
                                visual_requirements 基于 buildAvailableSignals 输出）
                    Editor   → Timeline v1
                               （标签召回 + 逐镜头 VLM 精选，
                                含候选镜头 + 未入选原因）
                    Critic   → Critique
                               （裁决：接受/修改/中止 + 补丁；
                                最多 2 轮迭代）
                    Render   → MP4（ffmpeg-kit，
                               可用时启用 h264_mediacodec）
```

## v5.0+ 旁路入口

### 自选入口（用户挑选素材 + 撰写意图）

```
CuratorScreen → UserCurationRequest → CurationRequestStore.stage
              → Worker (KEY_CURATION_REQUEST_ID)
              → runFromCuration:
                  UserCurationPlanner → synthetic Event(eventId="user_<requestId>")
                  IntentParserAgent.parseInitial → UserBrief
                  (parsedHook/payoff/tone/pace/duration/mustHaveSubjects/
                   parsedAvoid/captionPolicy/parsedColorGrade/rawText)
                  perception + annotation pass on selection
                  runEvent(userBrief=...) — 智能体在每个交接点应用覆盖参数
```

### 迭代入口（用户对已生成 vlog 给出反馈）

```
IterationSheet（故事板点击定位 + 文本框 + 4 个芯片）
→ IterationFeedback → IterationStore.stagePending
→ Worker (KEY_ITERATION_EVENT_ID)
→ iterate():
    if (text or targeted-shot non-empty):
      boot Gemma → IntentParserAgent.parseFeedback(timeline + raw)
                   → refined IterationFeedback with parsedScope
    dispatch by parsedScope:
      RENDER_ONLY → IterationPlanner.applyRenderOnly + render        (~30s)
      SHOT_LEVEL  → applyRevisions on cached timeline + render       (~30-60s)
      GLOBAL      → applyGlobalToDirector + buildTimeline rebuild
                     + applyGlobalToTimeline + render                (~2-3min)
      MIXED       → GLOBAL flow + SHOT_LEVEL revisions + render      (~3-4min)
    VersionArchive auto-archives prior MP4 →
      candidates/archive/<eventId>__<timestamp>.mp4
    IterationStore writes iter_<N>/ artifacts + appends history
```

## 关键设计决策

- **方法提取是负载承载的，而非装饰性的。** 如果内联，`PipelineOrchestrator.run()` 的协程状态机将超出 ART 的 64K 字节码限制，且从保存标签恢复的调度会中断（每次进度挂起都会从 `run()` 顶部重新启动，导致无限循环）。每个事件的主体放在 `runEvent()` 中；标注通道放在 `runAnnotationPass()` 中。**如果你添加另一个通道，请保持为独立的 suspend 函数。** 完整的调试故事见 [CHANGELOG.md v4.1](../CHANGELOG.md)。

- **两遍事件选择**（v4.4）。EventSelectionPlanner 排序两次——第一次基于轻量信号（资产数量、视频、GPS 分布、人脸、场景多样性、意图偏差）选出 top-K 候选；然后 EventScout 读取每个 top-K 事件的联系人表，Gemma 返回故事弧/人物/节奏信号。最终排序结合两者。仅靠粗排信号会混淆"孩子的生日派对"和"厨房早餐"；Scout 通道用每个候选约 3-5 秒的 VLM 调用消除歧义，结果 JSON 缓存。

- **缓存是逐资产的，两级**：轻量感知 JSON 位于 `filesDir/perception_cache/<assetId>.json`；VLM 标签在 Gemma 标注后 patched 进同一文件。EventScout 有自己的 `event_scout_cache/<eventId>.json`。第二次运行同一相册时跳过所有 VLM 工作。

- **节奏 + 调色从 Audience → Director → 渲染器 贯通。** Audience 输出 `pace` 枚举；Director 的 `target_duration_sec` 被限制在该节奏的区间内（紧凑 15-18s / 均衡 18-22s / 悠长 22-28s）。Director 还直接输出 `color_grade` 枚举，Editor 和渲染器无需从 `tone` 关键词推断。

- **Critic 使用补丁而非重写**（v4.4）。Critic 不再重新输出完整的 ShotRequest 对象，而是返回 `{shot_order, patches: {field: value}}` —— Token 占用更小，在 Gemma 4 E2B 上可靠性提升约 3 倍。裁决是显式的：`accept` 直接输出，`revise` 应用补丁，`abort` 写入面包屑并输出当前时间线（不再在死胡同事件上浪费 Critic 迭代）。

- **可用信号输入 Director。** Director 撰写镜头蓝图前，编排器会计算资产实际存在的场景分组 + 长视频 + 显著性摘要。Director 的提示词要求 `visual_requirements` 必须匹配真实可用的场景——避免在事件中不存在"举杯特写"这类资产时仍然 hallucinate 该镜头。

- **每次 VLM 调用超时**（90 秒，在 `AgentRuntime.ask` 中）：卡住的 Gemma 调用会返回部分文本，而不是永远挂起 Worker。
